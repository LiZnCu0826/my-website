# 疯狂策划 - 游戏架构总览

## 游戏概念

> 将经典烹饪后厨玩法完全置换为游戏策划职场模拟：
> - **厨师 → 游戏策划**（你）
> - **食客 → 四类玩家群体**（氪金大佬、白嫖党、休闲玩家、硬核玩家）
> - **做菜上菜 → 执行运营动作**（数值调整、活动运营、体验优化、内容更新）
> - **满意度 → 群体留存与口碑**
> - **店长订单 → 公司KPI考核**（流水、留存、好评）

## 核心系统架构

```
resources/         ← 数据层：玩家群体、运营动作、KPI定义
├── PlayerGroupType.gd      - 4类玩家群体枚举
├── GameAction.gd            - 运营动作Resource
├── ActionDatabase.gd        - 11种运营动作数据库
├── KpiDefinition.gd         - KPI任务Resource
└── KpiDatabase.gd           - 7天KPI任务数据库

autoload/          ← 系统层：全局单例
├── EventBus.gd              - 全局信号总线（跨场景通信）
├── KpiManager.gd            - KPI评估与惩罚管理
└── GameManager.gd           - 核心游戏循环（每天5回合 → 评估 → 下一天）

scenes/            ← 场景层：UI组件
├── MainGame.tscn            - 主游戏场景（入口）
├── GameOver.tscn            - 结算面板
└── UI/
    ├── SatisfactionBar.tscn - 单个玩家群体满意度条
    ├── KpiBar.tscn           - 单个KPI进度条
    ├── ActionCard.tscn       - 运营动作卡牌（可点击）
    ├── DayStartPanel.tscn    - 每日KPI简报
    └── DayEndPanel.tscn      - 每日KPI评估结果
```

## 游戏循环

```
天开始 → KPI简报（DayStartPanel）
  → 5个时间单位，每个单位选择一个运营动作
    → 动作影响各群体满意度 + KPI进度
    → 动作有冷却机制，不能连续使用
  → 时间耗尽 → KPI评估（DayEndPanel）
    → 未达标 → 累积压力值
    → 压力≥100 → 被辞退
  → 进入下一天 → 满意度自然衰减
  → 第7天结束 → 根据KPI通过数量给出最终评价
```

## 开发者笔记

- **类型安全**：所有信号携带强类型参数，所有变量显式类型标注
- **信号驱动**：EventBus 作为全局通信中枢，UI 组件通过信号解耦
- **组合优先**：SatisfactionBar / KpiBar / ActionCard 都是独立可复用的场景组件
- **资源数据**：GameAction 和 KpiDefinition 使用 Godot Resource 模式，可序列化、可在编辑器中创建
- **场景隔离**：每个 UI 面板都是独立 instancable 的 tscn
